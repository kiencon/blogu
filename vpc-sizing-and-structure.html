VPC Sizing and Structure Knowledge Takeaways

Introduction

This lesson focuses on designing a well-structured and scalable network inside AWS using a VPC. It emphasizes the importance of planning an IP plan for a business, especially when running in AWS.

Key Considerations for VPC Design

  • IP Range Selection: Deciding on the IP range (CIDR) for the VPC is crucial as it's not easy to change later.
  • VPC Size: The size of the VPC influences how many services can fit into it. Each service occupies space inside the VPC with one or more IPs.
  • Network Interaction: Consider other networks you'll interact with to avoid overlapping or duplicate ranges which could complicate communication.
  • Future Growth: Predict future needs and changes in the network structure to accommodate growth.

VPC Structure Breakdown

  • Tiers: IT networks typically have tiers like web, application, and database tiers. These tiers separate components and allow different security applications.
  • Availability Zones: Networks are often split across availability zones for resiliency.
  • Subnet Allocation: Each tier should have its own subnet in each Availability Zone. For example, four Availability Zones would mean four subnets per tier (web, app, DB, spare).

VPC Sizing Guidelines

  • Micro VPC: A /24 VPC with eight /27 subnets, totaling 216 IP addresses.
  • Extra Large VPC: A /16 VPC with sixteen /20 subnets, totaling over 65,000 IP addresses.
  • Subnet Calculation: If starting with a /16 VPC, splitting into 16 subnets results in /20 subnets, each with 4,091 IP addresses.

Designing for Animals4Life Organization

The design process considered the following:

  • Avoided Ranges: Specific ranges were avoided due to existing usage in other networks (e.g., 192.168.10.0/24, 10.0.0.0/16, 172.31.0.0/16).
  • Preferred Range: Used the 10.x.y.z range but avoided common subnets like 10.0 to 10.10.
  • Regional Allocation: Allocated specific /16 blocks for regions (US, Europe, Australia), ensuring non-overlapping ranges.
  • Account Segregation: Each AWS account (General, Prod, Dev, Reserved) received four /16 ranges per region, allowing for multiple VPCs.

Conclusion

Designing a VPC involves careful planning of IP ranges, considering current and future needs, avoiding overlaps, and structuring subnets based on tiers and availability zones. This structured approach ensures scalability and resilience in AWS environments.
